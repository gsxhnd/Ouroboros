# 10 - 设计决策与注意事项

## 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 架构模式 | Tauri 内嵌 HTTP server | 前端统一通信层，桌面/Web 复用同一前端代码 |
| API 协议 | REST + gRPC 双协议 | REST 用于常规操作（前端友好），gRPC 用于流式事件和高性能场景 |
| 数据库 | SQLite 为主，PostgreSQL 可选 | 嵌入式无需部署，服务化时可切换 |
| 文件策略 | 不转移文件 | 尊重用户文件组织习惯，减少数据风险 |
| 缩略图存储 | 资源库 .ourboros/thumbnails/ | 随资源库一起迁移、备份 |
| Rust HTTP | Axum | 性能优秀、生态成熟、与 Tokio 深度集成 |
| 前端状态 | zustand | 轻量、TypeScript 友好、无 boilerplate |
| 前端 UI | shadcn/ui | 可定制、无运行时依赖、Tailwind 集成 |
| 资源库模型 | 一实例一库，可切换 | 简化并发问题，桌面端通过多实例支持多库 |

## 注意事项

### 性能

1. **文件监听**：大型资源库（数万文件）需注意 watcher 的内存占用，考虑分级监听或延迟索引
2. **缩略图生成**：使用后台线程池异步生成，不阻塞主线程和 API 响应
3. **虚拟滚动**：前端必须使用虚拟列表，避免一次性渲染大量 DOM
4. **RAW 格式**：RAW 解码计算量大，考虑缓存解码结果

### 数据安全

5. **数据库迁移**：使用 migration 工具管理 schema 变更，确保向前兼容
6. **多实例安全**：SQLite WAL 模式处理并发，需确保同一资源库不被多个实例同时写入
7. **文件删除**：默认仅从数据库移除索引，需用户二次确认才删除实际文件

### 跨平台

8. **路径处理**：统一使用 `/` 分隔符存储，运行时按系统转换
9. **文件系统差异**：注意 macOS (APFS) / Windows (NTFS) / Linux (ext4) 对大小写敏感性差异
10. **gRPC-Web**：浏览器不支持原生 gRPC，需通过 `tonic-web` 层转换

### 架构

11. **数据库抽象**：通过 Rust trait 定义 Repository 接口，SQLite 和 PostgreSQL 各自实现
12. **错误处理**：core 层使用 `thiserror` 定义领域错误，server 层转换为 HTTP 状态码
13. **配置管理**：支持环境变量 + 配置文件双重来源，服务部署时通过环境变量覆盖
